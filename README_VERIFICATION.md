# Tiny-GPU UVM Verification Project

Этот проект представляет собой полноценное окружение верификации на базе методологии UVM (Universal Verification Methodology) для [Tiny-GPU]

> **Примечание:** Для данного GPU отсутствует строгая спецификация. Тем не менее, в ходе написания и запуска тестов были обнаружены архитектурные недочеты и логические баги, которые приводят к зависанию устройства или некорректному выполнению инструкций в определенных граничных условиях.



## Найденные аппаратные недочёты и баги (Bug Report)

Ниже представлен список обнаруженных проблем в RTL-коде GPU. 

### 1. Зависание планировщика на частичных блоках (Partial Blocks Deadlock)
**Файл:** `src/scheduler.sv`  
**Описание:** Планировщик жёстко зашит забирать следующий адрес инструкции (`next_pc`) у самого *последнего* потока в блоке, независимо от того, активен этот поток или нет. Если общее количество потоков не кратно размеру блока (4), последнему ядру достанется частичный блок. Выключенный поток в таком блоке всегда выдаёт `next_pc = 0`. В результате ядро принудительно прыгает на 0-й адрес, попадает в бесконечный цикл и никогда не выдаёт сигнал завершения.  
**Возможное решение:** Брать значение `next_pc` у нулевого потока в блоке (`current_pc <= next_pc[0]`), так как он гарантированно активен при любом размере блока.

### 2. Неработающий флаг Negative в АЛУ (Unsigned Comparison Bug)
**Файл:** `src/alu.sv`  
**Описание:** При выполнении инструкции `CMP` вычисляются флаги для условных переходов. Флаг отрицательного результата вычисляется как `(rs - rt < 0)`. Так как переменные `rs` и `rt` объявлены без спецификатора `signed`, стандарт языка Verilog расценивает выражение как беззнаковое. В беззнаковой арифметике результат вычитания не может быть меньше нуля, поэтому флаг `N` аппаратно всегда заклинен в `0`. Условный переход `BRn` (Branch if Negative) не сработает никогда.  
**Возможное решение:** Явно привести операнды к знаковому типу: `($signed(rs) - $signed(rt) < 0)`.

### 3. Залипание сигнала `done` (Start/Done Synchronization)
**Файл:** `src/dispatch.sv`  
**Описание:** Диспетчер выставляет `done <= 1` при `blocks_done == total_blocks`. Однако в коде нет механизма автоматического снятия этого сигнала при запуске нового ядра. Если после завершения первого запуска сразу начать второй (поднять `start`), сигнал `done` останется равен `1`. Тестовое окружение (или процессор-хост) решит, что второе ядро отработало мгновенно.  
**Возможное решение:** Добавить сброс флага `done <= 0` в блок `if (!start_execution)`.
